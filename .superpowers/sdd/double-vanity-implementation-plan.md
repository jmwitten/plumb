# DV72 double-sink vanity — implementation plan

**Contract:** `double-vanity-design.md`  
**Research:** `vanity-plumbing-research.md`,
`public-model-ingestion-research.md`

## Increment 1 — catalog reference boundary

1. Add failing schema/serialization tests for metadata-only
   `CatalogAssetRef`, licensing, units/frame/transform, calibration anchors,
   and consumer authority.
2. Implement the smallest immutable types and manifest serialization.
3. Prove that visual references cannot be consumed by cut-list, machining,
   plumbing, structural, or code-validation paths.
4. Prove an empty local asset cache leaves all construction truth unchanged.

## Increment 2 — analytic fixture, faucet, and jurisdiction adapters

1. Add K-20000 and K-T14414-4/K-410-K catalog facts with primary-source URLs,
   revision metadata, and distinct authority fields.
2. Add typed fixture, countertop/clamp, faucet, valve, water-target, and
   service envelopes.
3. Add versioned NYC 2022 and IPC 2024 lavatory profiles and pin their
   deliberate trap-distance difference.
4. Reject universal or uncited plumbing constants.

## Increment 3 — DV72 authoring and analytic model

1. Register opt-in `vanity.double_sink@1` and add compact
   `floating_double_sink_four_drawer@1` expansion.
2. Build a 72 in floating body with two equal structural/service bays, four
   fronts, two analytic basins, two faucet rough-ins, and two independent
   plumbing paths.
3. Derive one upper U-drawer and one shortened lower drawer per bay from the
   plumbing/service envelopes; reuse the cabinet pack's MOVENTO and drawer
   joinery contracts rather than copy their literals.
4. Generate closed, extended, removal, and service-state geometry.
5. Model a continuous rear structural rail/backing/anchor path while keeping
   the connection capacity unresolved.

## Increment 4 — validation and artifacts

1. Pin static and dynamic collision checks, service-opening dimensions,
   independent trap topology, fixture placement, runner applicability, and
   mounting representation.
2. Emit the nine required release gates with provenance and ensure release is
   blocked.
3. Build design-study inventory, unresolved procurement, coordination steps,
   and model-derived diagrams without inventing fabrication instructions.
4. Confirm old V36 behavior and all existing cabinet outputs are unchanged.

## Increment 5 — design-study document

1. Generate a self-contained HTML study with overall elevation/plan/section,
   per-bay plumbing/drawer interaction, motion/removal/service states, mount
   load path, code profile, and release-gate table.
2. Bind hover/display names to the canonical model labels.
3. Verify in the browser at desktop and narrow widths, inspect console output,
   and confirm no unstated values appear on reader surfaces.

## Gates and integration

1. Run focused schema/model/validation/document tests.
2. Ask a fresh adversarial reviewer to inspect geometry truth, plumbing/code
   claims, public-asset authority, release behavior, and V36 non-regression.
3. Fix Critical/Important findings and obtain confirmation.
4. Run one full suite on the final clean tree.
5. Commit in reviewable increments, fast-forward `main` only after verifying
   the remote head, push, regenerate from the accepted commit, and deliver the
   study beside the cabinet documents.
